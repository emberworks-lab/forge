# Git workflow conventions

Universal across projects. Default branch is `develop` if it exists, else `main`.

## Branch model

- **One branch per Linear epic.** Name: `feature/<prefix>-<N>-<slug>` (e.g. `feature/emb-227-combat-foundation`).
- **Sub-tickets are commits, not branches.** All sub-issues of an epic land on the same epic branch.
- **No long-lived feature branches outside this model.** Hotfixes use `hotfix/<slug>`; release PRs use `release/<version>`.
- **Branch creation is execution-time, not creation-time.** Don't `git checkout -b` inside `/create-epic` — the epic might never be picked up. Create the branch only when starting Story 1 of that epic.

## Commit model

- **One commit per sub-ticket.** Squash multiple WIP commits before posting to the branch (or use `git commit --fixup` + `git rebase --autosquash` discipline).
- **Magic-word commits link to Linear.** Format: `<MagicWord> <PREFIX>-<NUM>: <description>`.

| Magic word | Effect on PR merge to default | Use for |
|---|---|---|
| `Implements`, `Fixes`, `Closes`, `Resolves` | Auto-closes ticket | Feature work, bug fixes |
| `Refs`, `Part of`, `Related to`, `Contributes to` | Links only, no status change | Partial work, docs, refactor that touches the ticket but doesn't complete it |

- **One ticket per commit ideally.** Multi-ticket commits work but lose granularity: `Implements EMB-42, EMB-43: ...`.
- **Co-author footer.** End every commit message with the appropriate Co-Authored-By line for the executing model.

Examples:
```
Implements EMB-228: damage pipeline with Result<T> error returns

Refs EMB-227: extract turn order helper into formulas/

Fixes EMB-235: bloc test for combat error path
```

## PR model

- **PR is created AFTER manual testing.** Don't auto-create PR at epic close — manual test cases land in Linear epic comment first; user verifies; only then PR.
- **One PR per epic.** PR body links the epic, summarizes commits, references manual test cases comment.
- **Draft PR is OK** for early review while manual testing is ongoing — final marking ready-for-review happens after manual verification.

PR body skeleton (generated by `/pr-create`):
```markdown
Closes <epic ID> (linked in commits)

## Summary
<1-2 sentences from epic ## What>

## Commits
<list of magic-word commits, one per ticket>

## Manual verification
See Linear epic comment for full QA checklist.

🤖 Generated with [Claude Code](https://claude.com/claude-code)
```

## Squashing

- `/simplify-branch` runs as a step inside `/epic-close` (with explicit consent prompt) and may rewrite commits.
- After simplify, branch history should be clean: one commit per ticket, no fixups, no WIP, no merge commits from default branch (rebase, not merge).
- Force-push to the epic branch after simplify is expected (`git push --force-with-lease`).

## What NOT to do

- **No auto-commit without explicit user instruction**, EXCEPT inside `/execute-epic` where invocation = consent for per-ticket commits.
- **No branch creation inside `/create-epic`** or `/create-ticket` — those are creation skills, not execution skills.
- **No staging of secrets.** Never `git add` `.env*`, `*.key`, `*.pem`, credentials, tokens.
- **No merge commits from default into epic branch.** Rebase only.
- **No commits to default branch directly.** Even tiny fixes go through PR.

## Co-author footers

When a commit is made by Claude (any flavor), append the appropriate footer. Defaults the harness uses:

```
Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

The model running the commit owns the footer line. `/execute-epic` writes the footer matching the executor model that produced the work.
